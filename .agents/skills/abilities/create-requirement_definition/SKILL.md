# 要件定義書の作成

イントーク結果をもとに、機能要件・非機能要件・スコープを文書化する。

## やること / やらないこと

| やること | やらないこと |
|----------|-------------|
| 要件定義書の作成・更新 | 基本設計・詳細設計の記述 |
| スコープと受入条件の明確化 | API 仕様・DB スキーマの具体化 |
| 未決事項の明示 | コードの作成 |

設計の詳細は次段階（create-basic_design）に委ねる。

---

## 入力

次を読み込んでから執筆する。

| 入力 | パス / ソース |
|------|--------------|
| プロジェクト概要 | `projects/<project>/docs/README.md` |
| イントーク結果 | 会話履歴（ヒアリング / grill-me） |
| 前回版（差し戻し時） | `projects/<project>/docs/features/<feature>/requirements.md` |
| 修正指示 | レビュー指摘 or 人間からのフィードバック |

入力が不足している場合は、**仮定を明記したうえで** 文書を作成する。止まって質問しない（pm-agent がイントーク済みの前提）。

---

## 出力

**パス:** `projects/<project>/docs/features/<feature>/requirements.md`

**言語:** 日本語

---

## 執筆手順

1. 入力を読み、目的・スコープ・ユーザー像を整理する
2. [assets/requirements-template.md](assets/requirements-template.md) をベースに文書を作成する
3. 未決事項は「未決事項」セクションに列挙する（本文中に TODO を散らさない）
4. ファイルを保存する

### 差し戻し時

- 前回版をベースに、指摘・修正指示を反映した**全体版**を書き直す（差分だけ書かない）
- 対応した指摘は文書内で解消されていること

---

## 記述ガイドライン

### 機能要件（FR）

- **動作・振る舞い** を書く。「どう実装するか」は書かない
- 1要件 = 1つの検証可能な単位
- 優先度: `必須` / `任意` / `将来`

```
✅ FR-001: ユーザーは在庫数量を一覧で確認できる
❌ FR-001: 在庫テーブルに quantity カラムを持つ（設計の領域）
```

### ロール・権限

- ロールを定義し、**ロール × 操作** の権限マトリクスを作る
- 操作は機能要件 ID（FR-xxx）と対応づける
- アクセス制御が不要なら「該当なし（全ユーザー同一権限）」と明記する

```
✅ ROLE-002（閲覧者）は FR-002（在庫更新）が ×
❌ 権限はシステムが適切に制御する（曖昧）
```

### 非機能要件（NFR）

最低限、次を検討して記載する（該当なしは「該当なし」と明記）。

- 性能（レスポンス、同時接続など）
- セキュリティ（認証、認可、データ保護）
- 可用性・運用

### 受入条件（AC）

- 機能要件ごとに、**テスト可能な形** で書く
- 「〜できること」ではなく「〜のとき、〜であること」

```
✅ AC-001: 在庫一覧画面を開いたとき、登録済み商品が0件以上表示される
❌ AC-001: 在庫一覧が正しく動作する
```

### 未決事項（OPN）

- イントークで決まらなかった点を正直に列挙する
- 基本設計で解決すべきものと、人間判断が必要なものを区別しない（すべて列挙）

---

## 品質チェック（自己確認）

保存前に次を確認する。

- [ ] スコープの「含む」「含まない」が明記されている
- [ ] ロール・権限が定義されている（不要なら「該当なし」を明記）
- [ ] 機能要件が設計レベルに踏み込んでいない
- [ ] 受入条件が機能要件と対応している
- [ ] 未決事項が隠れていない
- [ ] 日本語で統一されている
